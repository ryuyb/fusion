# Fusion 直播聚合平台 - 技术架构

## 项目类型
Fusion 是一个**直播内容聚合后端服务**，采用 Go 语言开发，提供 RESTful API 服务，聚合多个直播平台（斗鱼、虎牙、哔哩哔哩、Twitch 等）的直播内容。

## 核心架构

### 架构模式
- **Clean Architecture** - 分层架构设计
- **Dependency Injection** - 依赖注入模式
- **Repository Pattern** - 数据访问抽象
- **Modular Monolith** - 模块化单体架构

### 架构分层
```
┌─────────────────────────────────────────┐
│          Interface Layer (HTTP)         │  ← 外部接口层
│  Handlers, Routes, Middleware, Swagger  │
├─────────────────────────────────────────┤
│         Application Layer               │  ← 应用层
│   Services, DTOs, Use Cases, Validators │
├─────────────────────────────────────────┤
│            Domain Layer                 │  ← 领域层
│  Entities, Repository Interfaces, Rules │
├─────────────────────────────────────────┤
│        Infrastructure Layer             │  ← 基础设施层
│ Database, Repositories, Config, Logger  │
└─────────────────────────────────────────┘
```

### 依赖方向
- **Interface** → **Application** (应用服务)
- **Application** → **Domain** (实体和接口)
- **Infrastructure** → **Domain** (实现仓储接口)
- **Domain** ← **Infrastructure** (无依赖)

## 外部平台集成

### 直播平台 API
- **斗鱼直播**: 开放 API
- **虎牙直播**: Web API
- **哔哩哔哩直播**: 开放平台
- **Twitch**: Helix API
- **其他平台**: 可扩展设计

### API 集成模式
- **统一接口**: 标准化的平台适配器
- **数据标准化**: 统一的直播数据格式
- **错误处理**: 平台级别的错误抽象
- **重试机制**: 指数退避、限流处理

### HTTP 客户端配置
每个直播平台都使用独立的 Resty v3 客户端实例：

```go
// 示例：平台 API 客户端配置
type PlatformClient struct {
    client  *resty.Client
    baseURL string
    apiKey  string
}

// 客户端特性配置
- 重试策略: 3 次重试，指数退避
- 超时设置: 连接超时 5s，读取超时 10s
- 请求日志: 记录所有外部 API 调用
- 错误处理: 统一错误格式
- 认证: 基于 API Key/Token
```

### 数据同步策略
- **拉取模式**: 定时拉取平台数据
- **Webhook**: 实时推送（部分平台支持）
- **轮询**: 直播状态变化检测
- **缓存**: Redis 缓存热点数据

### Resty v3 在平台集成中的使用
1. **统一认证**:
   - 每个平台配置专用的认证中间件
   - 自动添加 API Key、Token 等认证信息
   - 支持 OAuth2.0 流程（部分平台）

2. **重试和回退**:
   - 网络错误: 3 次重试，指数退避 (100ms, 200ms, 400ms)
   - 限流错误: 5 次重试，固定间隔 1s
   - 5xx 错误: 2 次重试，指数退避
   - 4xx 错误: 不重试，直接返回错误

3. **请求/响应中间件**:
   - 请求前: 添加认证信息、记录请求日志
   - 响应后: 记录响应时间、检查错误码
   - 错误处理: 统一错误格式和错误码

4. **性能优化**:
   - 连接池: 复用 TCP 连接
   - HTTP/2: 支持多路复用
   - 压缩: 启用 GZIP 压缩
   - 缓存: 智能缓存非敏感数据

5. **监控和日志**:
   - 请求日志: 记录所有 API 调用
   - 性能指标: 响应时间、成功率
   - 错误追踪: 错误率和错误类型统计
   - 告警: API 故障或性能下降告警

## 技术决策记录

### 1. Clean Architecture
**决策**: 采用 Clean Architecture 而非 MVC
**原因**: 业务逻辑与基础设施解耦，易于测试和维护
**替代方案**: MVC、洋葱架构
**权衡**: 学习曲线陡峭，但长期收益高

### 2. Fiber v3 vs Gin
**决策**: 选择 Fiber v3
**原因**: 性能优异、API 友好、中间件生态
**替代方案**: Gin、Echo
**权衡**: 相对较新，但性能更好

### 3. EntGO vs GORM
**决策**: 选择 EntGO
**原因**: 类型安全、代码生成、Graph 集成
**替代方案**: GORM、sqlx
**权衡**: 学习成本高，但开发效率和质量更高

### 4. uber-go/fx
**决策**: 使用 uber-go/fx 进行依赖注入
**原因**: 成熟稳定、与 Go 生态融合
**替代方案**: wire、手动依赖注入
**权衡**: 运行时开销极小

### 5. PostgreSQL vs MySQL
**决策**: 选择 PostgreSQL
**原因**: 丰富特性、JSON 支持、扩展性强
**替代方案**: MySQL
**权衡**: PostgreSQL 更适合复杂查询

### 6. Resty v3 vs 标准库 http.Client
**决策**: 选择 Resty v3
**原因**: 链式 API、重试机制、中间件支持
**替代方案**: 标准库 http.Client、go-retryablehttp
**权衡**: 更高抽象度，但依赖额外库

### 7. Resty v3 在平台集成中的优势
**特性对比**:
- 重试机制: 内置 vs 需要手动实现
- 请求/响应中间件: 支持 vs 不支持
- 链式调用: 友好 vs 繁琐
- 错误处理: 统一 vs 分散
- 性能: 优化 vs 需自行优化
