# User Notification System - 需求文档

## 1. 功能概述

用户通知系统旨在为 Fusion 直播聚合平台提供全面的通知能力，支持多种通知渠道（邮件、网页推送、Webhook），允许用户管理通知偏好和订阅主播的开播提醒状态，并提供完整的通知历史记录。系统将确保用户能够及时收到订阅主播的开播提醒，同时提供灵活的通知管理功能。

## 2. 用户故事

### 2.1 通知接收

#### 故事 1: 接收开播提醒邮件
- **优先级**: 高
- **作为** 已订阅主播的登录用户
- **我想要** 在订阅主播开播时收到邮件通知
- **以便** 及时了解喜欢的主播开播信息

**验收标准 (EARS)**:
```
Given 我已订阅某主播并开启了邮件通知和开播提醒
When 该主播在任意平台开播
Then 我应在5分钟内收到邮件通知
And 邮件包含主播名称、平台、开播时间
And 邮件包含直接观看链接
```

#### 故事 2: 接收网页推送通知
- **优先级**: 高
- **作为** 浏览器打开 Fusion 页面的用户
- **我想要** 收到浏览器原生推送通知
- **以便** 即使在不浏览 Fusion 时也能收到提醒

**验收标准 (EARS)**:
```
Given 我已授权浏览器推送通知并订阅了主播且开启开播提醒
When 该主播开播
Then 即使切换到其他标签页也应收到推送通知
And 点击通知可直达直播间
And 通知显示主播头像和平台信息
```

#### 故事 3: 配置 Webhook 通知
- **优先级**: 中
- **作为** 高级用户或开发者
- **我想要** 将开播通知推送到我指定的 Webhook URL
- **以便** 与我的其他系统集成（如 Discord、Slack、自建机器人）

**验收标准 (EARS)**:
```
Given 我已配置 Webhook URL并订阅了主播且开启开播提醒
When 该主播开播
Then 系统应 POST 通知数据到我的 Webhook URL
And 数据格式为 JSON 包含主播信息和直播链接
And 应支持重试机制（3次指数退避）
```

### 2.2 订阅主播开播提醒管理

#### 故事 4: 关闭单个主播的开播提醒
- **优先级**: 高
- **作为** 登录用户
- **我想要** 关闭特定已订阅主播的开播提醒
- **以便** 不想收到该主播的开播通知，但仍保持订阅关系

**验收标准 (EARS)**:
```
Given 我订阅了主播A、主播B、主播C
When 我在订阅列表中关闭主播A的开播提醒
Then 主播A的开播提醒应被禁用
But 我仍能看到主播A的在线状态
And 主播A开播时我不应收到任何通知
But 关闭提醒后再次开启应该可以正常收到通知
```

#### 故事 5: 开启单个主播的开播提醒
- **优先级**: 高
- **作为** 登录用户
- **我想要** 开启之前关闭的已订阅主播的开播提醒
- **以便** 重新开始接收该主播的开播通知

**验收标准 (EARS)**:
```
Given 我订阅了主播A且已关闭其开播提醒
When 我在订阅列表中开启主播A的开播提醒
Then 主播A的开播提醒应被启用
And 该主播下次开播时我应正常收到通知
And 通知设置应保持开启直到我再次关闭
```

#### 故事 6: 批量管理主播开播提醒
- **优先级**: 中
- **作为** 登录用户
- **我想要** 一次性开启或关闭多个主播的开播提醒
- **以便** 高效管理大量订阅的主播通知设置

**验收标准 (EARS)**:
```
Given 我订阅了多个主播，其中一些已开启、一些已关闭提醒
When 我使用批量操作功能选择多个主播并设置提醒状态
Then 所选主播的开播提醒状态应统一更新
And 操作完成后显示成功/失败结果
And 支持按状态筛选主播（全部/已开启/已关闭）
```

#### 故事 7: 查看主播提醒状态
- **优先级**: 中
- **作为** 登录用户
- **我想要** 在订阅列表中查看每个主播的开播提醒状态
- **以便** 快速了解哪些主播开启了提醒

**验收标准 (EARS)**:
```
Given 我订阅了多个主播
When 我查看订阅列表
Then 每个主播条目应显示开播提醒状态
And 状态应清晰标识（开启/关闭）
And 状态图标应直观易懂
And 支持按状态筛选和排序
```

### 2.3 通知偏好管理

#### 故事 8: 设置通知渠道偏好
- **优先级**: 高
- **作为** 登录用户
- **我想要** 选择我希望接收通知的渠道
- **以便** 只通过我喜欢的方式接收通知

**验收标准 (EARS)**:
```
Given 我已登录用户中心
When 我进入通知偏好设置
Then 我可以为每个渠道独立设置启用/禁用
And 我可以设置不同渠道的优先级
And 保存后设置立即生效
```

#### 故事 9: 设置通知时间偏好
- **优先级**: 中
- **作为** 登录用户
- **我想要** 设置免打扰时间段
- **以便** 在休息时间不被打扰

**验收标准 (EARS)**:
```
Given 我已设置 22:00-08:00 为免打扰时间
When 有主播在此时段开播且已开启提醒
Then 系统不应发送通知
But 应该在 08:00 后发送延迟通知
And 通知中标注为"延迟通知"
```

#### 故事 10: 按平台设置通知
- **优先级**: 中
- **作为** 登录用户
- **我想要** 选择只接收特定平台的通知
- **以便** 过滤掉我不关注的平台

**验收标准 (EARS)**:
```
Given 我只关注斗鱼和虎牙平台
When 我在偏好中只勾选这两个平台
Then 订阅其他平台主播时不应收到通知
And 已订阅其他平台主播的订阅应自动暂停提醒
```

### 2.4 通知历史

#### 故事 11: 查看通知历史
- **优先级**: 高
- **作为** 登录用户
- **我想要** 查看我收到的所有通知记录
- **以便** 了解历史通知情况或找回错过的重要通知

**验收标准 (EARS)**:
```
Given 我已收到多条通知
When 我进入通知历史页面
Then 我应看到按时间倒序的所有通知
And 每条记录显示：时间、渠道、主播、状态
And 支持分页浏览（每页20条）
```

#### 故事 12: 通知状态追踪
- **优先级**: 中
- **作为** 登录用户
- **我想要** 查看通知的发送状态
- **以便** 了解哪些通知成功送达，哪些失败

**验收标准 (EARS)**:
```
Given 我收到了一条通知
When 我查看通知详情
Then 应显示：发送时间、送达时间、渠道、状态
And 状态包括：已发送、已送达、已读、失败
And 失败时显示错误原因
```

#### 故事 13: 标记已读/未读
- **优先级**: 低
- **作为** 登录用户
- **我想要** 手动标记通知为已读或未读
- **以便** 管理未读通知数量

**验收标准 (EARS)**:
```
Given 我有未读通知
When 我点击某条通知的已读按钮
Then 该通知应标记为已读
And 未读计数应相应减少
And 支持批量标记为已读
```

## 3. 业务规则

### 3.1 通知触发规则
- 只有用户明确订阅且开启开播提醒的主播开播才发送通知
- 主播状态从离线变为在线时触发通知
- 同一主播的重复开播（5分钟内）不重复通知
- 通知延迟不应超过 5 分钟
- 关闭提醒的主播不会触发任何通知渠道

### 3.2 提醒状态规则
- 默认情况下，订阅主播时开播提醒为开启状态
- 用户可以随时开启/关闭任意已订阅主播的开播提醒
- 关闭提醒不影响订阅关系，仅影响通知发送
- 提醒状态变更立即生效
- 批量操作支持多选主播统一设置

### 3.3 渠道规则
- **邮件通知**: 必须配置有效邮箱地址，可配置 SMTP 服务器
- **网页推送**: 需要用户主动授权，支持桌面和移动浏览器
- **Webhook**: 需要有效 URL，响应状态码 2xx 视为成功
- 只有当全局渠道启用且主播提醒开启时才发送该渠道通知

### 3.4 频率限制
- 单个用户每分钟最多接收 5 条通知
- 相同主播的开播通知间隔至少 5 分钟
- Webhook 失败后重试最多 3 次，间隔为 30s, 60s, 120s
- 提醒状态变更操作每秒最多 10 次

### 3.5 数据保留
- 通知历史保留 90 天
- 用户偏好设置永久保留
- 提醒状态设置永久保留
- Webhook 失败日志保留 30 天

## 4. 非功能性需求

### 4.1 性能要求
- 通知发送延迟: < 1 分钟（P95）
- 提醒状态变更响应时间: < 200ms
- 并发处理能力: 1,000 通知/秒
- 系统可用性: 99.9%
- 邮件送达率: > 95%
- 推送送达率: > 90%

### 4.2 扩展性要求
- 支持水平扩展通知服务
- 通知队列可独立扩展
- 支持多区域部署
- 邮件服务商可多实例配置
- 提醒状态存储可独立扩展

### 4.3 安全要求
- Webhook 请求应包含签名验证
- 邮件发送需验证发件人身份
- 推送通知应防止伪造
- 敏感配置信息加密存储
- 提醒状态变更需要用户认证

### 4.4 可维护性
- 通知渠道可独立配置
- 支持动态开启/关闭渠道
- 支持动态开启/关闭主播提醒
- 统一的错误处理和日志记录
- 监控指标完善

## 5. 依赖项

### 5.1 内部系统
- **用户系统**: 获取用户信息、邮箱、偏好设置
- **订阅系统**: 订阅关系、订阅状态、主播信息
- **平台集成系统**: 主播开播信息、平台信息

### 5.2 外部服务
- **邮件服务**: SMTP 服务（如 SendGrid, Amazon SES）
- **推送服务**: 浏览器推送 API
- **监控系统**: 通知发送监控、告警

### 5.3 数据库
- **PostgreSQL**: 通知记录、用户偏好、提醒状态、模板配置
- **Redis**: 通知队列、缓存去重信息、提醒状态缓存

## 6. 约束条件

### 6.1 技术约束
- 必须遵循 Fusion 的 Clean Architecture
- 使用 uber-go/fx 进行依赖注入
- 邮件发送使用已配置的 SMTP
- 推送通知遵循 W3C Web Push 标准
- 提醒状态变更需要事务保证一致性

### 6.2 业务约束
- 不能影响用户隐私，用户可随时关闭通知
- 通知内容应简洁明了，避免垃圾信息
- 尊重用户的免打扰设置和提醒状态
- 遵守反垃圾邮件法规

### 6.3 资源约束
- 邮件服务有日发送量限制
- 推送通知有浏览器限制
- Webhook 调用有频率限制
- 提醒状态存储空间限制

## 7. 假设和前提

### 7.1 系统假设
- 用户已配置有效的邮箱地址
- 浏览器支持 Web Push API
- Webhook URL 稳定可用
- SMTP 服务已正确配置
- 已有完整的订阅管理系统

### 7.2 技术前提
- 已有用户认证系统
- 已有订阅管理系统
- 已有平台 API 集成
- 已有数据库基础设施
- 已有通知渠道基础实现

## 8. 风险和缓解

### 8.1 邮件送达风险
- **风险**: 邮件进入垃圾箱或被拒收
- **缓解**: 配置 SPF、DKIM、DMARC 记录，使用可靠邮件服务商

### 8.2 推送通知风险
- **风险**: 浏览器权限或兼容性问题
- **缓解**: 提供备选通知渠道，逐步引导用户授权

### 8.3 Webhook 风险
- **风险**: Webhook 服务不稳定或响应慢
- **缓解**: 重试机制、超时控制、失败告警

### 8.4 性能风险
- **风险**: 大量通知导致系统过载
- **缓解**: 队列机制、限流、熔断器

### 8.5 数据一致性风险
- **风险**: 提醒状态变更导致数据不一致
- **缓解**: 事务控制、幂等操作、最终一致性
